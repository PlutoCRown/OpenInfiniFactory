# 开发规范
1. 与你开发的用户是不懂 Rust 的人，如果他提出了不 Rust style 的做法，你应该先指出让其确认。
1. 正常情况下，你无需执行 cargo check 和 cargo run。除非用户指明你需要实际运行一下试试，你才在当前会话中运行一次。
2. 做完较大修改，你可以运行一下 `bun scripts/log_rs_lines.js` 然后告知用户本次更新完之后，整体代码行数变更了多少。提交前必须运行一次，然后向用户报告。
3. 提交时，提交信息应该包含用户在提出这些修改的主要目的，而非代码更改了什么。
4. 如果用户提出的需求并非更新游戏逻辑，而是优化代码或系统结构。那么你必须先评估该优化是否能减少代码量，或者使得结构更通用，并且使得系统的维护难度降低，然后你再去执行。
5. 永远不要写 #[allow(unused_imports)] 和 #[allow(dead_code)]，没用的东西就直接删掉。
6. 游戏运行的模拟期间需要执行的逻辑要严格按照 ECS 规范开发，但是游戏中的 UI 应该根据 React 的风格开发。


# 项目状态
1. 该项目是一个多平台游戏，计划涵盖多种桌面端、移动端甚至 Web 平台，当涉及到平台特性的时候，注意它们的兼容层。例如涉及输入输出设备的 API。
1. 当前项目处于早期状态，你可以做出重大变更，并且不考虑兼容之前的旧数据。
2. 如果你认为项目当前的结构是不合理的，那么它就是不合理的。你可以提出本次需要进行重大重构，然后执行。

# 和 InfiniFactory 不一样的地方。
1. 方块传感器现在只会检测为平台方块和材料方块，而非所有方块。
2. 新增吸盘、折射镜等功能。方块传感器还会检测激光。
3. 方块检测器不会给相邻的用电器供电。
4. 永远不会做计数器。
