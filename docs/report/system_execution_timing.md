# 系统执行时机报告

本文说明当前游戏什么时候执行哪些逻辑，以及 Bevy 的 `Update` 调度和“真正做重活”的区别。

## 结论

`add_systems(Update, ...)` 的意思是系统会被 Bevy 放进每帧调度。这些函数每帧都会被调用一次，但很多系统开头会检查游戏模式、UI 状态、输入事件或模拟状态；条件不满足时会很快返回。

因此需要区分两层：

- 调度频率：Bevy 是否每帧调用该 system。
- 实际工作：该 system 在本帧是否真的修改世界、重建渲染、执行模拟或处理 UI。

## 启动时执行

启动阶段注册在 `src/game/mod.rs` 的 `Startup`：


| 系统                               | 作用                   |
| -------------------------------- | -------------------- |
| `setup_scene`                    | 创建相机、灯光、地面 / 初始渲染资源。 |
| `spawn_player`                   | 创建玩家 / 飞行相机控制实体。     |
| `refresh_saves_on_startup`       | 扫描保存列表。              |
| `ui::load_ui_font`               | 加载 UI 字体资源。          |
| `ui::setup_ui`                   | 构建 HUD、菜单、面板等 UI 实体。 |
| `systems::debug::setup_debug_ui` | 创建调试 UI。             |


这些只在进入游戏应用时执行一次。

## 每帧调度

主插件把大部分游戏系统注册在 `Update`。这表示每帧调度一次，执行顺序大致是：


| 顺序  | 系统组                                                            | 说明                                |
| --- | -------------------------------------------------------------- | --------------------------------- |
| 1   | `camera_move`、`camera_look`、`gameplay_input`、`placement_input` | 玩家移动、视角、基础输入、放置 / 删除 / 旋转。        |
| 2   | 菜单 action 系统                                                   | 主菜单、保存列表、暂停菜单、方块配置面板等按钮处理。        |
| 3   | `simulation_controls`、`tick_simulation`                        | 模拟开始 / 暂停 / 单步 / 回滚，以及模拟回合 tick。  |
| 4   | `apply_fov`、`update_hover`、`draw_hover_structure_bounds`       | 相机参数、瞄准目标、结构线框。                   |
| 5   | `animate_blocks`                                               | 方块和活塞动画插值。                        |
| 6   | UI 更新系统                                                        | 状态栏、按钮 hover、面板可见性、背包、保存列表、配置文本等。 |
| 7   | debug 系统                                                       | 调试开关、性能 UI、玩家碰撞框等。                |


性能统计系统 `mark_perf_*` 也在这些阶段之间运行，用来测量每段耗时。

## 每帧但通常早退

以下系统每帧调度，但不会每帧都做重活：


| 系统                    | 早退条件                                | 真正做事的时机                         |
| --------------------- | ----------------------------------- | ------------------------------- |
| `gameplay_input`      | UI 阻塞、当前不是可操作模式等                    | 按 Esc、打开 / 关闭背包或暂停菜单、切换游戏模式。    |
| `placement_input`     | 不在 Playing、UI 阻塞、模拟正在运行             | 鼠标点击放置 / 删除、按键旋转、切换方块形态、拖拽选择区域。 |
| 菜单 action 系统          | 当前 `GameMode` 或面板不匹配、按钮没有 `Pressed` | 用户点击菜单、保存列表、暂停菜单、方块设置面板。        |
| `simulation_controls` | 不是 Play 模式、不是 Playing、UI 阻塞         | 按模拟、单步、快速、回滚按键。                 |
| `tick_simulation`     | 不在 Play，或者模拟未运行且没有单步请求              | 模拟运行中累计时间到一回合，或收到单步请求。          |
| `update_hover`        | UI 阻塞或无目标                           | 更新当前瞄准方块和 hover 结构包围盒。          |
| UI 更新系统               | 对应面板不可见或数据未变化                       | 面板打开、文本变化、按钮 hover、滚动 / 拖动。     |
| debug 系统              | debug 未打开或按键未触发                     | 切换调试显示、更新调试文本、绘制调试 gizmo。       |


## 玩家放置 / 编辑方块时执行

放置相关入口是每帧调度的 `placement_input`，但只有输入事件触发时才会修改世界。

主要编辑动作：


| 动作       | 主要函数                  | 后续同步                                   |
| -------- | --------------------- | -------------------------------------- |
| 放置 / 删除  | `commit_edit_gesture` | 修改 `WorldBlocks`，刷新静态虚拟方块，重建工厂结构，重建渲染。 |
| 旋转方块     | `rotate_block_at`     | 更新 facing，刷新静态虚拟方块，重建工厂结构，播放旋转动画或重建渲染。 |
| 切换方块形态   | `alternate_block_at`  | 切换 block kind，刷新静态虚拟方块，重建工厂结构，重建渲染。    |
| 移动选择区域   | `move_selection`      | 移动选中方块和焊接关系，重建工厂结构，必要时重建渲染。            |
| 拾取方块到快捷栏 | `pick_target_block`   | 只改 UI / 快捷栏选择，不改世界。                    |


这里的“刷新静态虚拟方块”是 `refresh_static_generated_markers`，用于生成焊点、钻头头部等不依赖信号的派生方块。它不是每帧执行，而是在编辑动作、加载世界、回滚 / 重置等明确世界变化点执行。

## 加载 / 切换世界时执行

加载世界入口主要在 `src/game/systems/menus.rs`：


| 场景      | 主要函数                       | 后续同步                                                 |
| ------- | -------------------------- | ---------------------------------------------------- |
| 从保存列表加载 | `open_loaded_world`        | `load_world` 替换 `WorldBlocks`，刷新静态虚拟方块，重建工厂结构，重建渲染。  |
| 清空并回主菜单 | `clear_loaded_world`       | 清空世界，清状态，重建空世界渲染。                                    |
| 从解法切回编辑 | `switch_to_edit_mode`      | 恢复 puzzle snapshot，刷新静态虚拟方块，切换背包和模式。                 |
| 重置解法    | `reset_solution_world` 调用处 | 恢复 puzzle snapshot，刷新静态虚拟方块，清模拟状态，重建结构和渲染。           |
| 模拟回滚    | `rollback_simulation` 调用处  | 恢复 start snapshot 或删除材料，刷新静态虚拟方块，清 pending 生成物，重建渲染。 |


这些都是事件触发，不是每帧执行。

## 模拟回合执行

模拟回合入口是 `tick_simulation`。它每帧调度，但只有两种情况会调用 `run_turn`：

- 玩家请求单步。
- 模拟正在运行，并且累积时间达到一个回合。

`run_turn` 是一个完整模拟回合。当前内部大致顺序是：


| 顺序  | 阶段           | 说明                                       |
| --- | ------------ | ---------------------------------------- |
| 1   | 清理 / 放置到期生成物 | 清理派生 marker，把上一回合准备好的生成物放入世界。            |
| 2   | 重力标记         | 对材料结构和活动工厂结构生成垂直移动计划。                    |
| 3   | 信号检查         | 刷新信号网络，计算通电组件和通电设备。                      |
| 4   | 通电 marker    | 生成阻拦器等依赖通电状态的派生碰撞。                       |
| 5   | 设备移动标记       | 传送带、抬升器、活塞、旋转器等生成结构移动计划。                 |
| 6   | 合并并执行移动      | 按移动标记优先级合并，递归推动结构，统一执行移动并生成动画。           |
| 7   | 重新生成 marker  | 移动后刷新静态 marker 和通电 marker。               |
| 8   | 材料行为         | 验收、焊接、切削、标记、转换、传送等。                      |
| 9   | 准备下一回合生成     | 计算下一回合的 generator preview / pending 生成物。 |
| 10  | 信号刷新和渲染重建    | 为通电电线颜色和动画重建运行期渲染实体。                     |


更细的回合内部细节见 `docs/report/simulation_turn_phases.md`。

## 每帧渲染 / 动画相关

渲染实体不会每帧完整重建。完整重建通常发生在：

- 加载 / 清空世界。
- 放置、删除、旋转、切换方块。
- 模拟回合执行完，需要用新世界状态和动画重建。
- 切换结构调试显示。

每帧真正持续运行的是 `animate_blocks`，它只更新已经带有 `AnimatedBlock` 或 `AnimatedPusher` 组件的实体插值；动画结束后会移除对应组件。

pending generator preview 是例外：`tick_simulation` 在 Play 模式下会刷新预览实体，用当前 accumulator 表示生成进度。这是为了让生成器预览动画能在模拟回合之间连续显示。

## 性能含义

当前有不少系统注册在 `Update`，这是 Bevy 的常规写法。它不一定意味着每帧都执行昂贵逻辑。

需要注意的高成本操作主要是：

- 遍历整个 `WorldBlocks`。
- 重建 `FactoryStructureState`。
- `despawn_world` 后重建所有渲染实体。
- 刷新信号网络。
- 刷新派生 marker。

这些操作当前主要集中在编辑动作、加载 / 回滚、模拟回合结束等明确时机。它们不应该被无条件放到每帧路径里。如果以后新增功能，应优先接入这些事件入口，避免在普通 `Update` 系统里每帧全量扫描世界。

## 建议

后续可以把一些“每帧调度但事件触发”的逻辑拆成更明确的事件系统，例如：

- `WorldTopologyChanged`
- `WorldLoaded`
- `SimulationTurnCompleted`
- `DebugModeChanged`

这样可以进一步减少手动调用刷新函数的遗漏风险，也能让系统执行时机更容易从代码里看出来。
